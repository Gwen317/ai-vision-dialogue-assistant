# AI 视觉对话助手：技术选型与多维论证报告 (Tech Stack Evaluation)

在构建“AI 视觉对话助手”这样高频、流式、低延迟的双工多模态系统时，技术栈的选型将直接决定系统的**首字延迟 (TTFT)**、**带宽消耗**、**计算资源的分配合理性（端侧 vs 云侧）**以及**工程落地速度**。

本报告对核心技术组件进行了全方位的比选与论证。

---

## 一、 前端核心框架比选 (Frontend Core)

| 选项 | 优势 (Pros) | 劣势 (Cons) | 结论与选型依据 |
| :--- | :--- | :--- | :--- |
| **React + Vite** *(首选)* | 1. 声明式 UI 完美适应复杂的异步状态流（VAD、Socket、Player、Canvas等）。<br>2. React Hooks (Ref/State) 便于维护全局音视频上下文与状态机实例。<br>3. Vite 热更新极快，现代生态（如 Vis.js/D3.js 绑定）极其丰富。 | 框架体积相对较大，有 React 渲染周期的额外开销。 | **选用**：长连接双工对话状态繁杂，React 的组件化与强状态管理是保障代码不成为“面条代码”的基础。 |
| **Vue 3 + Vite** | 1. 响应式系统性能极佳。<br>2. SFC (单文件组件) 开发体验友好。 | 多模态音视频等非 DOM 操作的逻辑中，Vue 的 Proxy 包装可能带来不必要的开销，Ref 操作与原生 JS 接口对接稍显繁琐。 | **备选**：与 React 处于同一级别，若团队熟悉 Vue 可等价替代。 |
| **Vanilla JS (纯原生)** | 零框架开销，极简体积，运行速度最快。 | 当存在 FSM（状态机）迁移、拓扑图谱渲染、AR 坐标转换时，DOM 手动更新逻辑会变得臃肿不堪，难以维护。 | **放弃**：仅适合写单页 Demo，不适合具有复杂交互的系统。 |

## 二、 后端网关技术栈比选 (Backend Gateway)

本系统的后端本质上是**媒体流网关（Media Gateway）**，而非传统的增删改查业务服务器。它需要承载二进制音频分片拼接、并发 WebSocket 连接、大模型 SSE 流转发以及 Abort 控制信号。

| 评估维度 | Node.js (TypeScript) *(首选)* | Python (FastAPI) | Java (Spring Boot / WebFlux) |
| :--- | :--- | :--- | :--- |
| **网络 I/O 性能** | **极高**：单线程事件循环机制完美契合高并发 WebSocket 媒体中转。 | **高**：基于 `asyncio` 和 `uvicorn`，适合中转。 | **高**：采用 Reactive WebFlux (Project Reactor) 支持非阻塞。 |
| **音视频流处理** | **极简**：支持 Buffer 直接读写、裁剪，直接使用 `Buffer.concat` 拼接分片，零文件存储开销。 | **极简**：利用 `bytes` 处理方便，流操作简单。 | **复杂**：Java 处理 byte 数组拼接和流转较为繁琐，且对 Opus、WebM 等非标准音频的解析依赖沉重。 |
| **AI 生态与 SDK** | **极佳**：Google Gen AI 官方 Node.js SDK 迭代极快，对流式推理（Stream）和多模态 `inlineData` 支持极其完美。 | **顶级**：AI 算法本源生态，调用 CLIP Embedding 方便。 | **滞后**：Spring AI 等框架对最新多模态 Gemini API 的双工流式接口支持滞后，开发成本高。 |
| **打断机制实现** | **极简**：原生支持 `AbortController` 绑定 Socket 事件，一键强杀云端大模型拉取线程。 | **简单**：利用 `asyncio.Task.cancel` 取消协程。 | **较难**：响应式流的取消逻辑较为曲折，容易产生死锁或流泄露。 |
| **结论** | **选用**：作为高性能媒体网关，Node.js 是最敏捷、资源占用最少且对 WebSocket 流处理最友好的技术栈。 | **备选**：如果项目包含大量“本地离线深度学习推理”（如本地跑大模型/本地 CLIP），可转用 Python。 | **放弃**：对于实时流式双工网关，Java 开发效率低，I/O 框架偏重。 |

## 三、 本地边缘 AI 引擎选型 (Edge-AI Inference)

我们设计了本地 VAD（过滤静音）和端侧 YAMNet（分类噪音）。这些模型必须直接在用户的浏览器里运行，以最大化降低云端成本。

* **选型结论**：**`ONNX Runtime Web` + `Wasm`** 结合 **`TensorFlow.js`**。
* **多维论证**：
  1. **ONNX Runtime Web (Wasm 引擎)**：
     - **作用**：运行语音活动检测模型（Silero VAD）。
     - **论证**：Silero VAD 官方提供了 ONNX 格式模型。ONNX Runtime Web 能够利用 **WebAssembly (Wasm)** 调用多线程和 SIMD 指令，在前端 CPU 上仅需 1-2% 的开销即可实现毫秒级的声音分类判定。
  2. **TensorFlow.js**：
     - **作用**：运行 YAMNet 噪声分类。
     - **论证**：Google 官方将 YAMNet 精简版托管在 TFHub，并为 TF.js 提供了现成的加载接口，可以零开销获取特征。通过 WebGL/WebGPU 加速，可以在端侧 100ms 内判定声音是否为人声，防止无效触发。

## 四、 音视频通信协议选型 (Network & Streaming)

如何将音视频发送给网关？有三种协议路线：

```
[路线 1] 客户端 MediaRecorder ──(200ms WebM Opus 分片)──> WebSocket (主线推荐)
[路线 2] 客户端 WebRTC MediaStream ────────────────────> WebRTC Gateway (超低延迟)
[路线 3] 客户端 HTTP POST ─────────────────────────────> REST API (传统API)
```

1. **路线 1：WebSocket 二进制切片帧 (选用)**
   - **机制**：前端使用 HTML5 `MediaRecorder` 每 200ms 抛出一个 WebM (Opus 编码) 的 `Blob`，读取为 `ArrayBuffer` 后通过 WebSocket 直接发送。
   - **论证**：WebM Opus 是浏览器原生支持的压缩编码，**无需前端加载额外的 WASM Opus 压缩包**，直接压缩了 10 倍带宽，上行仅需 24kbps。WebSocket 支持全双工，打断信号和音视频数据在同一个 TCP 管道内按序传输，绝对不会出现“打断信号比音频帧滞后”的问题。
2. **路线 2：WebRTC 双轨道传输 (高延迟优化版备选)**
   - **机制**：客户端与网关建立 WebRTC 连接，通过 RTP 协议传输实时音视频轨道。
   - **论证**：WebRTC 是低延迟的王者（<100ms）。然而，搭建 WebRTC 网关极其复杂，需要处理 STUN/TURN NAT 穿透、网关媒体协议转换（WebRTC 转标准 Stream 喂给 Gemini API）。在原型开发阶段，性价比极低。
3. **路线 3：HTTP Chunked / SSE (半双工放弃)**
   - **机制**：客户端通过常规 HTTP POST 提交，服务端用 Server-Sent Events 流式返回。
   - **论证**：HTTP 是单向的，在 AI 说话时，客户端无法在同一个信道内发送打断信号。打断必须通过另一个 HTTP 请求发起，此时在网络抖动时可能导致打断滞后，体验不佳。

## 五、 多模态情景记忆库 (Episodic Vector DB)

如何低成本、低负担地存储并检索用户的“视觉与文本情景记忆卡片”？

* **选型结论**：**`本地内存检索` -> 升级为 `Qdrant (嵌入式/本地Docker)`**。
* **多维论证**：
  1. **本地内存向量检索 (In-Memory Vector Store)**：
     - **机制**：将 CLIP 视觉向量和文本语义向量存储在 Node.js 后端的内存数组中，使用自定义余弦相似度算法进行循环扫描计算。
     - **论证**：在开发调试与初期 Demo 演示中，历史记忆卡片一般不会超过 1000 条。对于 1000 条以内的 512 维向量，JS 内存余弦相似度计算仅需不到 **1毫秒** 即可完成，**不需要任何复杂的数据库依赖**，部署极为简单，且性能极佳。
  2. **Qdrant (持久化与高并发拓展)**：
     - **机制**：一旦项目进入工业测试，转用轻量级向量库 Qdrant。
     - **论证**：Qdrant 拥有极低的内存占用和超快的 Rust 运行速度。它支持通过 Docker 一键部署，并且提供了完美的多维特征过滤支持（例如：在计算向量相似度时，可以同时通过时间戳进行过滤过滤过滤，仅检索 3 天内的记忆）。
