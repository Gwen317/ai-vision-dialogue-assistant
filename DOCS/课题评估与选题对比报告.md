# AI 课题对比与评估报告

本报告旨在对**课题一（AI 视觉对话助手）**与**课题二（AI 语音绘图工具）**进行深度对比与可行性分析，帮助明确研发方向、技术架构及最终呈现效果。

---

## 摘要与最终推荐

> [!TIP]
> **推荐选择：课题一（AI 视觉对话助手）**
> - **原因**：课题一具有极强的**视觉冲击力（WOW 效果）**和**实用场景**。结合 WebRTC、HTML5 Canvas 和大语言模型（如 Gemini Multimodal API），能够低延迟实现实时的“见你所见，听你所言”。此外，关于“端云协同成本控制”的工程设计空间非常大，易于写出深度十足的设计文档。
> - **相比之下**，课题二（语音绘图）虽然技术上有挑战性，但人机交互体验天然存在瓶颈（用语音指挥画笔进行高精度定位和绘制非常繁琐、延迟高，容易让用户感到挫败），视觉呈现和交互流畅度较难达到惊艳的效果。

## 课题一：AI 视觉对话助手 详细评估

### 1. 核心用户故事 (User Stories)
* **故事 A（实时环境感知）**：用户向摄像头展示一本书、一个手办或身处的房间，AI 能够立刻认出并围绕其展开交谈。
* **故事 B（双工语音对话）**：用户无需点击按钮，直接说话，系统通过端侧 VAD（语音活动检测）自动断句并流式传输语音，AI 实时进行语音回答。
* **故事 C（情境交互助手）**：用户展示一段手写代码或一道数学题，AI 实时圈出关键步骤并进行语音讲解。

### 2. 控制运营成本的技巧 (Cost Control & Edge-Cloud)
在多模态（视频流 + 语音）交互中，Token 消耗和 API 调用成本是最大的痛点。以下是可落地的优化技巧：
* **技巧 1：端侧关键帧提取 (Keyframe Filtering)**：
  - **方法**：不采用持续的高帧率视频传输。在前端使用 Canvas 计算相邻帧的像素差异度（SSIM 或简单差值）。只有当画面发生明显变化（如物体移入、大范围晃动停止后稳定）时，才发送当前帧作为视觉上下文。
  - **效果**：降低 80% 以上的视觉 Token 消耗。
* **技巧 2：端侧语音活动检测 (Local VAD)**：
  - **方法**：使用轻量级 Web 端的 VAD 库（如 `@ricky0123/vad-web` 或简单的 Web Audio API 能量门限检测）。
  - **效果**：仅在用户说话时才开启音频捕获并向云端发送，避免静音期和环境杂音造成的无效计费。
* **技巧 3：分辨率与帧率自适应压缩**：
  - **方法**：将摄像头采集的高清画面（如 1080p）在端侧使用 Canvas 缩放到 512x512 或更低，并压缩为 WebP/JPEG 格式后再传输。
  - **效果**：单张图片 Token 极低，传输延迟大幅下降。

### 3. 视觉设计与交互体验 (Aesthetic & Interaction)
* **科技感 UI**：采用暗黑科技风/赛博朋克界面。左侧为流式摄像头预览，带有时髦的动态扫描线和人脸/物体识别框；右侧为 AI 对话的历史气泡与状态指示器。
* **动效反馈**：底部集成 Web Audio API 渲染的**实时音频波形图 (Audio Visualizer)**，当用户说话或 AI 说话时，波形随频率跳动，提供强烈的交互反馈。

## 课题二：AI 语音绘图工具 详细评估

### 1. 核心指令能力 (Command Capabilities)
* **基础指令**：`“画一个圆形”`、`“填充红色”`、`“移动到中心”`。
* **复杂指令拆解**：`“在画布左上角画一栋小房子，烟囱里冒着烟”` -> 拆解为：画矩形（墙）、画三角形（屋顶）、画小矩形（烟囱）、画曲线（烟雾）。
* **容错与修正**：`“撤销上一步操作”`、`“把刚才的圆形变大一倍”`。

### 2. 交互瓶颈与局限性
* **空间定位困难**：没有鼠标/键盘，用户很难用语音精确描述位置（例如：“往左移动 30 像素，不对，再往右一点点”）。这会导致极高的交互延迟和用户挫败感。
* **指令理解容错性**：中文语音识别的同音字（如“圆”和“源”、“画”和“花”）在绘图上下文中容易出错。
* **效果上限受限**：如果仅支持几何图形绘制，界面容易显得像儿童画板，难以达到“WOW”的视觉效果；如果通过 API 异步生成 AI 图片（Text-to-Image），则变成了一个普通的 Prompt 绘图框，失去了“实时绘图过程”的交互乐趣。

## 两者多维度对比矩阵

| 评估维度 | 课题一：AI 视觉对话助手 (推荐 ⭐⭐⭐⭐⭐) | 课题二：AI 语音绘图工具 (评分 ⭐⭐⭐) |
| :--- | :--- | :--- |
| **技术实现复杂度** | **中等**：Web API 权限获取（音视频）+ 多模态大模型 API 对接。端侧优化是亮点。 | **高**：需要设计一套强大的 DSL（领域特定语言）或 Agent 来拆解复杂绘图指令并控制 Canvas。 |
| **用户体验与流畅度** | **极佳**：说话、展示物品，AI 就能实时互动，自然度高。 | **较差**：受限于语音定位的低带宽，操作繁琐、延迟明显。 |
| **视觉呈现 (WOW 效果)** | **极佳**：摄像头动态画面 + 实时波形图 + 识别动效 + 智能语音交互。 | **一般**：Canvas 几何图形绘制或静态 AI 图片生成，交互感偏弱。 |
| **运营成本控制空间** | **极大**：端侧 VAD、关键帧筛选、图片压缩、多模态 Token 管理，可体现深厚的架构设计。 | **较小**：主要是 STT API 和绘图生成 API 的开销，优化空间有限。 |
| **文档深度与学术价值** | **高**：端云协同架构设计、双工对话状态机、流量与成本优化算法。 | **中**：指令解析语法树、容错状态机。 |

## 下一步行动建议

1. **确认选题**：建议优先选择**课题一：AI 视觉对话助手**。
2. **技术栈选型**：
   - **前端**：Vanilla HTML5 + CSS（暗黑科技感 UI、CSS Gradients/Animations）+ vanilla JS / React / Vite。
   - **后端 / 云端**：Node.js 或 Java (对应你当前 workspace 的 Maven 项目) 搭建轻量级网关，用于调用大模型多模态接口（例如 Gemini / OpenAI GPT-4o ），或直接在前端进行安全代理请求。
   - **核心库**：`Web Audio API`（音频波形）、`Canvas API`（视频帧处理与压缩）、轻量级端侧 VAD（如 `onnxruntime-web` 驱动的 VAD）。
